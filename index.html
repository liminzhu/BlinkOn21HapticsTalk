What about the web?

These same interactions on the web? Very likely no haptics.

The only tool available today:

navigator.vibrate([200, 100, 200]);
  • Mobile-only — no Safari/WebKit support
  • Raw duration/pattern arrays — no semantic intent
  • No declarative path — always requires JS
  • Developers hand-tune millisecond values per device

Developers are already hacking around this

  • Safari never implemented navigator.vibrate()
  • But Safari's <input switch> fires a system haptic on toggle
  • So… check/uncheck invisible switches to get haptics

Goals & Non-goals

Goals

  • Semantic effects — signal intent, not raw patterns
  • Cross-platform — desktop & mobile
  • Declarative (CSS) + imperative (JS)
  • Progressive enhancement
  • Minimal fingerprinting surface

Non-goals

  • Identical output across platforms
  • Notification haptics (e.g. alert on task complete)
  • Gamepad / specialized hardware

The API at a glance

Four semantic effects — hint · edge · tick · align — with optional intensity (0.0–1.0).

Declarative (CSS)

@haptic-trigger button:active {
  effect: align;
  intensity: 0.8;
}

Fires when a selector starts matching.
No JavaScript needed.

Imperative (JS)

navigator.playHaptics('align', 0.8);

For interactions that need runtime logic.
Requires sticky user activation.

Effect vocabulary

EffectDescriptionWhen to use
hint Light, subtle cue Focusing an input, entering a drop zone
edge Heavy boundary signal Validation failure, scroll hitting boundary
tick Firm discrete pulse Scroll-snap landing, toggling a switch
align Crisp lock-into-place Drag-to-snap, window snap, zoom to 100%

intensity: normalized 0.0–1.0 · effective = system setting × developer value

Imperative API (JS)

navigator.playHaptics(effect, intensity);
  • effect: "hint" | "edge" | "tick" | "align"
  • intensity (optional): 0.0–1.0, defaults to 1.0
  • Requires sticky user activation — no permission prompt
  • Always returns undefined — no device info leakage

Feature detection:

if ('playHaptics' in navigator) { … }

Declarative API (CSS)

@haptic-trigger <selector> {
  effect: <effect-name>;
  intensity: <number>;       /* optional, defaults to 1.0 */
}
  • Fires once when selector starts matching
  • Works with pseudo-classes (:active, :checked, :snapped), classes (.is-open), attributes ([aria-invalid])
  • UAs may coalesce or suppress triggers to avoid overload
  • Progressive enhancement — browsers silently ignore unknown at-rules
Scenario

E-Commerce "Add to Cart"

Tactile confirmation on button press — zero JavaScript:

@haptic-trigger .add-to-cart:active {
  effect: align;
  intensity: 0.8;
}
Scenario

Flipping through albums

A tick each time an album snaps into view, using :snapped from CSS Scroll Snap 2:

@haptic-trigger .album-carousel > .album:snapped {
  effect: tick;
  intensity: 0.5;
}
Scenario

Drag-to-snap divider

CSS can't express distance-threshold logic — imperative API:

divider.addEventListener('pointermove', (e) => {
  const width = e.clientX;
  if (!snapped && Math.abs(width - SNAP_POINT) < 4) {
    snapped = true;
    navigator.playHaptics?.('align', 0.8);
  } else if (snapped && Math.abs(width - SNAP_POINT) >= 4) {
    snapped = false;
  }
});

Privacy & Security

Privacy

  • No device enumeration
  • No effect/capability queries
  • No success signal returned
  • No new media features in v1

Security

  • Activation model
    • Imperative: sticky user activation
    • Declarative (direct interaction): fires freely
    • Declarative (script-initiated): requires activation
  • Anti-abuse
    • No lasting effect — user can navigate away at any time
    • UA may throttle/suppress per-origin

Future extensions

  • Animation haptics — haptic-effect / haptic-intensity in @keyframes for multi-step choreography
  • Custom effects — developer-defined or platform-specific beyond the four builtins
  • prefers-haptics media feature — coarse user preference signal
  • Trigger phase control — exit / both in addition to enter-only (e.g. haptic on sidebar open and close)
@keyframes bounce-settle {
  0%   { transform: translateY(-100%); }
  40%  { haptic-effect: edge;  transform: translateY(0); }
  100% { haptic-effect: align; haptic-intensity: 0.5;
         transform: translateY(0); }
}
Design Decision

Which declarative model?

A. Selector-trigger proposed

@haptic-trigger button:active {
  effect: align;
}
  • Covers pseudo-classes, classes, and attributes — handles any app state
  • New spec concept; no CSS precedent for selector-match-as-trigger

B. Pseudo-class property

button:active {
  haptic-feedback: align;
}
  • Simplest syntax for built-in interactions
  • Cannot fire on class/attribute changes — misses most JS-driven app state

C. Transition-coupled

.btn { transition: scale 0ms;
       transition-haptic-effect: align; }
.btn:active { scale: 1; }
  • Familiar transition-* pattern; natural bidirectional firing
  • Non-animated interactions (e.g. button tap) need fake zero-duration transitions

D. Animation-trigger

@haptic tap { effect: align; }
button:active {
  haptic-name: tap;
}
  • @keyframes-like pattern; reusable named effects
  • Repeated same-action (e.g. two taps) needs value reset to re-trigger

Q: Should selector-trigger (A) anchor standardization, or is another model a better fit?

Quick-fire Questions

More to discuss

  • Activation model: split or uniform?
    Direct interaction fires freely; script-initiated requires sticky activation — or gate everything uniformly?
  • Effect vocabulary
    Are hint, edge, tick, align the right four? Do they map well to native primitives?
  • Should playHaptics return success?
    Useful for debugging — but risks leaking hardware info
  • :snapped dependency
    Acceptable for v1, or need a scroll-snap-haptic fallback property?
  • Promote any future extension to v1?
    Animation haptics? Custom effects? prefers-haptics media feature?
  • Global arbitration
    Standardize cross-element winner beyond per-element dedupe?
  • Device enumeration interest?
    Richer experiences vs. fingerprinting trade-off — worth exploring?

Appendix

JavaScript alternatives considered

  • Extending navigator.vibrate() — bolting semantic effects onto a pattern-based API is awkward; feature detection breaks; carries abuse stigma
  • Pointer-event based API (our earlier proposal) — purely imperative, no declarative path, ties haptics to specific input events
  • HTML attributes — requires one attribute per interaction type, can't compose with cascade or media queries
