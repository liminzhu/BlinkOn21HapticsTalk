Goals & Non-goals

Goals

  • Semantic effects — signal intent, not raw patterns
  • Cross-platform — desktop & mobile
  • Declarative (CSS) + imperative (JS)
  • Progressive enhancement
  • Minimal fingerprinting surface

Non-goals

  • Identical output across platforms
  • Notification haptics (e.g. alert on task complete)
  • Gamepad / specialized hardware

The API at a glance

Four semantic effects — hint · edge · tick · align — with optional intensity (0.0–1.0).

Both paths target the most recent input device — if not haptics-capable, no-op (no rerouting).

Declarative (CSS)

button:active {
  @haptic align 0.8;
}

Nested at-rule fires when parent selector starts matching.
No JavaScript needed.

Imperative (JS)

navigator.playHaptics('align', 0.8);

For interactions that need runtime logic.
Requires sticky user activation.

Effect vocabulary

EffectDescriptionWhen to use
hint Light, subtle cue Focusing an input, entering a drop zone
edge Heavy boundary signal Validation failure, scroll hitting boundary
tick Firm discrete pulse Scroll-snap landing, toggling a switch
align Crisp lock-into-place Drag-to-snap, window snap, zoom to 100%

intensity: normalized 0.0–1.0 · effective = system setting × developer value

Imperative API (JS)

navigator.playHaptics(effect, intensity);
  • effect: "hint" | "edge" | "tick" | "align"
  • intensity (optional): 0.0–1.0, defaults to 1.0
  • Requires sticky user activation — no permission prompt
  • Always returns undefined — no device info leakage

Declarative API (CSS)

<selector> {
  /* visual declarations */
  @haptic <effect-name> <intensity>?;
}
  • Fires once when parent selector starts matching — not on initial style computation
  • Works with any selector — pseudo-classes, classes, attributes
  • Per-rule tracking — two rules with the same effect both fire independently, no same-value collision
  • Script-initiated matches (e.g. classList.add()) require sticky user activation
  • Also nests inside @keyframes for multi-step haptic choreography
Scenario

E-Commerce "Add to Cart"

Tactile confirmation on button press — zero JavaScript:

.add-to-cart:active {
  scale: 0.95;
  @haptic align 0.8;
}
Scenario

Photo Carousel

A tick each time a photo snaps into view, using :snapped from CSS Scroll Snap 2:

.carousel > .photo:snapped {
  @haptic tick 0.5;
}
Scenario

Drag-to-snap divider

CSS can't express distance-threshold logic — imperative API:

divider.addEventListener('pointermove', (e) => {
  const width = e.clientX;
  if (!snapped && Math.abs(width - SNAP_POINT) < 4) {
    snapped = true;
    navigator.playHaptics?.('align', 0.8);
  } else if (snapped && Math.abs(width - SNAP_POINT) >= 4) {
    snapped = false;
  }
});

Future extensions

  • Custom effects — developer-defined or platform-specific beyond the four builtins
  • prefers-haptics media feature — coarse user preference signal
  • Trigger phase control — exit / both in addition to enter-only (e.g. haptic on sidebar open and close)

Privacy & Security

Privacy

  • No device enumeration
  • No effect/capability queries
  • No success signal returned
  • No new media features in v1

Security

  • Activation model
    • Imperative: sticky user activation
    • Declarative (direct interaction): fires freely
    • Declarative (script-initiated): requires activation
  • Anti-abuse
    • No lasting effect — user can navigate away at any time
    • UA may throttle/suppress per-origin

Design Decision

Which declarative model?

Haptics are one-shot, fire-and-forget — not ongoing state. We evaluated five models:

Model Syntax Syntax–semantics match Co-located Concise Re-trigger safe
A. Nested @haptic proposed button:active { scale: .95; @haptic align; } ✅ says "haptic" ✅ ✅ ✅ per-rule tracking
B. Standalone @haptic-trigger @haptic-trigger button:active { effect: align; } ✅ says "haptic" ❌ separate block ❌ two blocks ✅ per-rule tracking
C. Computed-value property button:active { haptic-feedback: align; } ❌ looks like state ✅ ✅ ❌ value collision
D. Animation-trigger @haptic tap { effect: align; }
button:active { haptic-name: tap; }		 ❌ looks like state ⚠️ split ❌ define+attach ⚠️ unique names
E. Transition-coupled .btn { transition: scale 0ms;
  transition-haptic-effect: align; }		 ⚠️ natural with transition ✅ ❌ needs fake transition ⚠️ dummy props

Syntax–semantics match = dev sees syntax, matches intuition  ·  Co-located = haptic in same rule block  ·  Concise = minimal syntax  ·  Re-trigger safe = same effect fires from distinct state changes

Q: Which tradeoffs best serve developers? Should nested @haptic anchor standardization?

Open Questions

Seeds for discussion

These are open questions in the explainer — happy to dive into any of these or wherever the conversation goes.

  • Activation model: split or uniform?
    Direct interaction fires freely; script-initiated requires sticky activation — or gate everything uniformly?
  • Effect vocabulary
    Are hint, edge, tick, align the right four? Do they map well to native primitives?
  • :snapped dependency
    Acceptable for v1, or need a scroll-snap-haptic fallback property?
  • Should playHaptics return success?
    Useful for debugging — but risks leaking hardware info
  • Device enumeration interest?
    Richer experiences vs. fingerprinting trade-off — worth exploring?

Appendix

JavaScript alternatives considered

  • Extending navigator.vibrate() — bolting semantic effects onto a pattern-based API is awkward; feature detection breaks; carries abuse stigma
  • Pointer-event based API (our earlier proposal) — purely imperative, no declarative path, ties haptics to specific input events
  • HTML attributes — requires one attribute per interaction type, can't compose with cascade or media queries
